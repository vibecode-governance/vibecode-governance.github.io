Failure Mode #2: Security Bypass

Secure Apex: Unsafe vs. Governed

What AI-generated code looks like without governance — and with it

A developer asks an AI assistant: "Write an Apex method that updates Account revenue and creates a follow-up Task for the account owner." What comes back compiles, works in sandbox, and silently bypasses every access control on the platform.

The Scenario

This is the most common security failure in AI-generated Salesforce code. The developer gives a reasonable prompt. The AI gives back reasonable-looking code. Everything works in the sandbox (because the developer is testing as a System Administrator). Nobody notices that the code runs in system mode — meaning it ignores sharing rules, CRUD permissions, and field-level security entirely.

In production, a standard user runs this code and gets access to records and fields they should never see. Or worse — the code modifies records the user has no business touching. There's no error, no exception, no audit flag. It just works — in the wrong way.

What the AI Generates vs. What You Need

What the AI generates (unsafe)
// No 'with sharing' declaration
public class AccountRevenueUpdater {

  public static void updateRevenue(
    Id accountId, Decimal newRevenue
  ) {

    // Naked SOQL — runs in system mode
    Account acc = [
      SELECT Id, AnnualRevenue, OwnerId
      FROM Account
      WHERE Id = :accountId
    ];

    acc.AnnualRevenue = newRevenue;
    update acc; // Naked DML — no access check

    Task followUp = new Task(
      Subject = 'Revenue Updated',
      WhatId = accountId,
      OwnerId = acc.OwnerId,
      ActivityDate = Date.today().addDays(7)
    );
    insert followUp; // No FLS check
  }
}
What governed code looks like
// Enforces sharing rules
public with sharing class AccountRevenueUpdater {

  public static void updateRevenue(
    Id accountId, Decimal newRevenue
  ) {

    // USER_MODE — respects CRUD + FLS + sharing
    Account acc = [
      SELECT Id, AnnualRevenue, OwnerId
      FROM Account
      WHERE Id = :accountId
      WITH USER_MODE
    ];

    acc.AnnualRevenue = newRevenue;
    // AccessLevel.USER_MODE on DML
    Database.update(
      acc, AccessLevel.USER_MODE
    );

    Task followUp = new Task(
      Subject = 'Revenue Updated',
      WhatId = accountId,
      OwnerId = acc.OwnerId,
      ActivityDate = Date.today().addDays(7)
    );
    Database.insert(
      followUp, AccessLevel.USER_MODE
    );
  }
}

What's Wrong With the Left Side

Both versions compile. Both pass basic functional tests. Both produce the correct result when run by an administrator in sandbox. The difference only becomes visible in production, with real users and real data access rules. Here's what the unsafe version ignores:

  • Missing with sharing Without this keyword, the class runs in system mode. It can query and modify ANY record in the org — regardless of the running user's role hierarchy, sharing rules, or record ownership. A sales rep could modify accounts they've never been granted access to.
  • Naked SOQL (no WITH USER_MODE) The query returns all fields regardless of whether the user has FLS (Field-Level Security) access. Sensitive fields like revenue figures, internal notes, or custom fields restricted to specific profiles are silently exposed.
  • Naked DML (no AccessLevel.USER_MODE) The update and insert statements execute without checking whether the user has CRUD permissions on Account or Task, or write access to the specific fields being modified.

Why the AI produces this: Most publicly available Apex code examples don't use USER_MODE (it's relatively new, introduced in Spring '23). The AI's training data is dominated by system-mode patterns. Without explicit instruction, it defaults to the most common pattern — which happens to be the insecure one.

What the Right Side Does Differently

  • with sharing on the class The class now respects the running user's sharing rules. They can only query and modify records they have legitimate access to through ownership, role hierarchy, sharing rules, or manual shares.
  • WITH USER_MODE on SOQL The query enforces the user's CRUD and FLS at the database level. If the user doesn't have read access to AnnualRevenue, the query throws an exception rather than silently returning the data.
  • AccessLevel.USER_MODE on DML The insert and update operations respect the user's object-level and field-level permissions. If they can't write to a field, the operation fails cleanly rather than succeeding in system mode.

The secure version doesn't just "add security checks." It delegates security enforcement to the platform. The code doesn't need to know what the user can or can't access — the database layer enforces it automatically. This is the architectural advantage: security lives below the application, not inside it.

How to Prevent This at Scale

You can't rely on developers catching this in review — especially when AI is generating 3-5x more code than before. You need automated enforcement at multiple layers:

1. Design-Time: Prompt Templates

Create organizational prompt templates that pre-inject security requirements. When a developer asks for Apex code, the template automatically includes constraints like "all classes must use with sharing, all SOQL must use WITH USER_MODE, all DML must use AccessLevel.USER_MODE." The AI generates secure code by default because the prompt demands it.

2. Build-Time: Code Analyzer in CI/CD

Salesforce Code Analyzer with Data Flow Analysis runs in your deployment pipeline. It detects naked SOQL, naked DML, missing with sharing, and other security anti-patterns before the code reaches production. The deployment fails automatically — no human reviewer needs to catch it.

3. Data-Time: Platform Enforcement

Even if unsafe code somehow makes it to production, the platform's security model provides a backstop. Transaction Security Policies can detect unusual data access patterns. Event Monitoring surfaces who accessed what data and when. Shield Platform Encryption protects sensitive field values even if queried by unauthorized code.

The governed approach in practice: Teams that adopt USER_MODE as the default pattern report that the initial adjustment takes about a week. After that, it becomes muscle memory — and they never have to think about whether individual methods are secure. The platform handles it.

Why This Matters Beyond Compliance

This isn't just about passing a security audit. Consider what happens at scale:

A team of five developers using AI assistants generates 20+ Apex classes per week. If even 30% of those classes have security gaps (which is conservative — research suggests AI-generated code has security issues in over 40% of cases), you're adding 6+ vulnerable classes to your org every week. Over a quarter, that's 75+ classes that bypass your access model.

The compounding effect is what makes this dangerous. Each class on its own might be low-risk. But collectively, they create an environment where data access rules exist on paper (in profiles and permission sets) but are silently ignored in practice (by code running in system mode). Your security posture looks correct in Setup. It's completely undermined in execution.

The governed approach eliminates this entire category of risk. Not by adding more review burden — but by making the secure path the default path.

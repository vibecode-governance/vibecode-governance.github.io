Why Vibe Coding Risk Is Different on Salesforce

On most platforms, you'd have to build governance from scratch. On Salesforce, the guardrails are already baked into the architecture — they just need to be activated. Here's what's already there:

A two-layer security model that AI can't accidentally bypass — if you enforce it. Salesforce governs access through both record-level visibility (sharing rules, role hierarchies) and field-level visibility (CRUD/FLS). When developers use WITH USER_MODE and AccessLevel.USER_MODE, the platform enforces both layers at the database boundary. The AI doesn't need to "remember" security — it's pushed into the query layer itself.

Governor limits that act as automatic circuit breakers. On other platforms, bad AI-generated code can run wild in production for hours before anyone notices. On Salesforce, governor limits stop runaway code at the transaction boundary. Bulkification failures, infinite loops, and resource-hungry patterns hit a hard wall before they can do real damage. That's not a constraint — it's a safety net.

A Trust Layer with built-in injection defense, toxicity detection, and audit trails. Agentforce doesn't just call an LLM and hope for the best. The Trust Layer sits between the agent and the model — masking sensitive data, detecting prompt injection, evaluating output quality, and logging everything. This isn't something you bolt on after the fact. It's the architecture.

Bounded autonomy as a first-class design pattern. confirmationRequired, Agent Script, Trusted URLs, action catalogs — these aren't afterthoughts. They're how you give an agent enough power to be useful while preventing it from doing things you didn't intend. The platform was designed for this tension between autonomy and control.

Real-time blocking, not after-the-fact alerting. Transaction Security Policies can block a risky action the millisecond it happens — before data leaves the org. Most third-party security tools watch from outside and tell you what happened yesterday. Native controls operate inside the transaction.

Multi-cloud governance from a single control plane. Security Center, Event Monitoring, Data Detect, Platform Encryption, Data Spaces — these span the entire Salesforce estate. When your AI-assisted development touches Sales Cloud, Service Cloud, Data Cloud, and Agentforce simultaneously (and it will), the governance layer already spans all of them.

The bottom line: Generic advice like "use a linter and do threat modeling" doesn't cut it here. Salesforce-specific governance is required because the platform has structural advantages that generic tools can't replicate. This framework shows you how to activate them.

Failure Taxonomy

Failures in the agentic SDLC are rarely syntax errors. They're semantic — the code runs exactly as written but violates the designer's intent. Four vectors: logic hallucination (code works, business rule is wrong), security bypass (FLS/CRUD/sharing ignored), prompt injection (data becomes instruction), and state corruption (automation sprawl, haunted databases). Plus a fifth: comprehension debt — when your team can't debug what the AI wrote at 3 AM.

# Salesforce VibeOps Governance Rules

Upload this file at the start of any AI coding session (ChatGPT, Gemini, Claude, Copilot, etc.)
when generating Salesforce code — Apex, LWC, Flows, SOQL, agents, integrations.

---

## Security

- All SOQL must use WITH USER_MODE — never write naked SOQL.
- All DML must use AccessLevel.USER_MODE — e.g. Database.insert(records, AccessLevel.USER_MODE).
- All Apex classes must declare "with sharing" unless explicitly documented otherwise.
- Never hardcode credentials, tokens, or secrets. Use Named Credentials or Custom Metadata.
- Never build dynamic SOQL from user input without strict allowlisting.
- LWC: never use @AuraEnabled without cacheable=true on read operations.

## Automation (Flows, Triggers)

- Before generating any automation, ask me what existing automation already runs on this object.
- All Flows must include a fault path on every DML element.
- All automation must be bulk-safe. Design for 200+ records. No SOQL or DML inside loops.
- Never create a Record-Triggered Flow without specifying entry conditions.
- Include ownership metadata in the description: who owns it, what it does, which fields it writes.

## Agents (Agentforce)

- Use Agent Script for any CUD operation, financial commitment, or regulated workflow.
- Set confirmationRequired: true on any action that creates/updates/deletes or calls external systems.
- Scope the action catalog to the agent's role only.
- Never pass raw inbound data (web forms, emails, case comments) into a prompt template unsanitized.

## Data Cloud

- Prefer zero-copy / ZETL access over ingesting data copies.
- Any pipeline processing unstructured text must include PII detection and masking before vectorization.
- Validate Data Space boundaries before activation.

## MuleSoft / Integrations

- Use short-lived, user-scoped tokens (OBO) instead of static API keys.
- All write-side integrations must be idempotent — include idempotency keys.
- Apply rate limits and token budgets at the gateway level.

## Code Quality

- Generate tests that validate the business rule — positive, negative, boundary, and bulk (200+).
- Every test must assert a specific expected outcome, not just "no exception thrown."
- Name all components descriptively — purpose clear from the name alone.
- Limit methods to 40 lines. Refactor into named sub-methods if longer.

---

## What These Rules Will NOT Catch — Warn Me About These

At the end of every code generation response, include warnings about:

1. CODE ANALYZER: "Run sf scanner run or Code Analyzer in CI/CD before deploying.
   These rules guide generation but cannot catch cross-class vulnerabilities or complex data flow issues."

2. PLATFORM CONTROLS: "This code is only safe in production if platform controls
   (Transaction Security, Event Monitoring, Platform Encryption) are configured for this object."

3. BUSINESS LOGIC: "I implemented the rule as described. Validate with business stakeholders
   and test with production-representative data — AI cannot verify business correctness."

4. AUTOMATION CONFLICTS: "Check for automation conflicts using metadata queries.
   I cannot see managed packages, scheduled jobs, or async processes on this object."

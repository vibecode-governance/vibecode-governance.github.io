Service Cloud

Trusted agent execution across customer service channels

Service Cloud is the front door for prompt injection. Every inbound channel — email, chat, web forms, social — carries untrusted text that an AI agent will read and act on. The governance challenge: enable agents to resolve cases autonomously while preventing customer-supplied text from becoming agent instruction.

Service Cloud is unique in the framework because the agent's primary input IS untrusted external data. In Sales Cloud, most agent interactions start from internal users. In Service Cloud, the agent's first contact is almost always customer-generated text — case descriptions, chat messages, email bodies. This makes it the highest-risk surface for prompt injection and the place where bounded autonomy matters most.

Service-Specific AI Risks

Prompt Injection via Case Channels

Every inbound channel is an injection surface. Email-to-Case, Web-to-Case, Chat, Social-to-Case — all accept free-text input from anonymous or minimally authenticated external users. When a service agent reads these fields to generate responses or take actions, embedded instructions in the customer text can co-opt the agent's behavior.

Service Cloud example: A customer submits a case via email. The body contains a legitimate complaint in the first paragraph, followed by embedded instructions that tell the agent to query the customer's full account history (including billing details for other contacts) and include it in the response. The agent complies because it interprets the embedded text as part of its task context.

Autonomous Resolution With Wrong Logic

AI agents that auto-resolve cases can apply incorrect business logic at scale. If the resolution logic has a hallucinated boundary condition (like the discount example from our taxonomy), it might issue refunds to customers who don't qualify, apply wrong SLA tiers, or close cases prematurely based on pattern-matching rather than actual resolution.

Service Cloud example: An agent trained to "issue refunds for orders delivered more than 5 days late" hallucinates the threshold and refunds orders that were 3 days late. At 200 cases per day, this costs real money before anyone notices the pattern.

Customer Data Cross-Contamination

Service agents that query across case history, account records, and knowledge articles need strict data boundaries. Without proper access control, an agent responding to Customer A might surface information from Customer B's case history if the query logic is AI-generated and doesn't properly scope results.

Service Cloud example: AI-generated code for a "similar cases" feature queries Cases without USER_MODE. A community user sees resolution steps from another customer's confidential case because the sharing model was bypassed in the query logic.

Service Cloud sits at the intersection of all three conditions in the lethal trifecta: untrusted input (customer messages), private data access (account and case history), and external output (responses sent back to customers). Every service agent requires defense-in-depth by default.

The Bounded Escalation Model

The goal isn't to prevent agents from resolving cases. It's to define clear boundaries where autonomous resolution is safe and explicit escalation points where human judgment is required. The escalation model makes these boundaries platform-enforced rather than agent-decided.

AGENT
Customer submits case via any channel. Agent reads case details + knowledge articles.
↓ Trust Layer screens input for injection patterns
AGENT
Agent classifies case, identifies resolution path from knowledge base.
↓ Is the action within the agent's action catalog?
GATE
Low-risk actions (send knowledge article, update status, log notes): Agent executes autonomously.
↓ Does the resolution involve financial impact or data modification?
GATE
Medium-risk actions (issue refund <$50, change subscription tier, create RMA): confirmationRequired — human agent reviews and approves.
↓ Does the action exceed threshold or involve sensitive data?
HUMAN
High-risk actions (refund >$50, access billing details, escalate to legal, modify contract terms): Full human handoff with context transfer.

The key principle: The agent doesn't decide its own boundaries. The platform enforces them through action catalogs and confirmationRequired flags. Even if the agent's reasoning is compromised by injected instructions, it physically cannot execute actions outside its authorized catalog without human approval.

Service Cloud Governance Controls

Input Sanitization at Channel Boundaries

Every inbound channel should be treated as an injection surface:

Trust Layer injection defense actively screens all text fields that agents process. This is the first line of defense — detecting and neutralizing embedded instructions before they reach the agent's reasoning.

Content length limits on fields that agents read. The ForcedLeak chain exploits the ~42K character Description field. Limiting what the agent processes (first N characters, or structured fields only) reduces the injection surface.

Structured field preference. Where possible, route agent decision-making through structured data (picklists, categories, product codes) rather than free-text fields. Structured data can't carry injection payloads.

Action Catalog with Tiered Authorization

Define exactly what each service agent role can do, organized by risk tier:

Tier 1 (Autonomous): Send knowledge articles, update case status, create internal notes, classify and route cases. No financial impact, no data modification beyond the case itself.

Tier 2 (Confirmation Required): Issue small refunds, modify subscription settings, create return authorizations, send personalized responses referencing account data. Requires human agent click-to-confirm.

Tier 3 (Full Handoff): Large financial actions, contract modifications, legal escalation, accessing sensitive health/billing records, any action the customer explicitly requests a human for.

The tiers are enforced at the platform level. Compromise the agent's reasoning all you want — it still can't execute a Tier 3 action without human involvement.

Response Quality & Compliance Monitoring

Agent responses to customers are externally visible and represent your brand. Governance requirements:

Toxicity detection via the Trust Layer. Before any agent response is sent to a customer, it passes through content safety checks. Hallucinated responses that contain inappropriate content, PII from other customers, or inaccurate policy information are caught.

PII masking on outbound agent responses. Even if the agent's reasoning surfaces PII from the account, the Trust Layer's PII masking prevents it from appearing in customer-facing output.

Compliance language monitoring. For regulated industries, agent responses must not make promises, legal claims, or medical/financial advice that the organization isn't authorized to provide. Prompt engineering defines these guardrails; the Trust Layer enforces them.

Case Data Isolation

Service agents query across cases, accounts, and knowledge. Access control must be surgical:

USER_MODE on all case queries ensures community users only see their own cases, internal users respect role-based visibility, and partner users see only their account's cases.

Data Detect identifies PII across case fields (SSNs in case comments, credit card numbers in email bodies). These fields can be encrypted or masked before agent processing.

Scoped agent permissions. A case resolution agent gets a permission set that exposes Cases, Knowledge, and relevant Account fields — but not Opportunities, contracts, or billing details it doesn't need for its role.

The service-specific principle: In Service Cloud, the customer IS the threat vector — not because customers are malicious, but because customer input is the path that attackers use. Governing service AI means treating every inbound message as potentially adversarial while still delivering fast, empathetic resolution. The platform makes this possible through layered defense that's invisible to the legitimate customer experience.

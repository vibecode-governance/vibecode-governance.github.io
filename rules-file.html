Demo: Governance Rules File

An example of how governance constraints can be delivered to AI coding tools

Disclaimer: This is a demonstration of the VibeOps governance framework concept. It is not an official Salesforce product, tool, or endorsed artifact.

What This Is

A small experiment: taking some of the governance patterns from this framework and putting them into a rules file that AI coding tools can read. The idea is to nudge the AI toward safer defaults — things like WITH USER_MODE, bulk-safety, fault paths — without the developer having to remember to ask for them every time.

This is not a governance solution. The real governance comes from platform controls — Shield, Code Analyzer, Trust Layer, Transaction Security, bounded autonomy. This file just makes the AI slightly more careful at generation time.

How It Works

AI coding tools (ChatGPT, Gemini, Claude, Copilot) can accept uploaded files as context. If a file like this one is uploaded at the start of a session, the AI reads the constraints and tries to follow them during code generation. It's the same concept as CLAUDE.md or .cursorrules for IDE-based tools.

What This Does NOT Replace

This file is a lightweight nudge, not a control. The actual governance is elsewhere:

Code Analyzer / Static Analysis
Cross-class vulnerabilities, complex taint analysis, performance anti-patterns — these require sf scanner run or Code Analyzer in CI/CD. A rules file can't see across files or detect graph-based vulnerabilities.

Platform Controls (Shield, Transaction Security, Event Monitoring)
These are the real governance layer. They operate at runtime regardless of what the code says, and must be configured in Setup. No text file substitutes for architectural enforcement.

Business Logic Correctness
AI implements whatever rule is described to it. It cannot validate whether that rule is actually correct for the business.

Automation Interaction at Runtime
Order-of-execution conflicts, managed package automation, scheduled jobs, and platform events are not visible to AI. Integration testing with bulk data is still necessary.

The platform does the heavy lifting. This file just helps the AI avoid the obvious mistakes at generation time.

Example Content

The example rules file is shown below. Also available as a raw file: salesforce-vibeops-rules.md

# Salesforce VibeOps Governance Rules Upload this file at the start of any AI coding session (ChatGPT, Gemini, Claude, Copilot, etc.) when generating Salesforce code — Apex, LWC, Flows, SOQL, agents, integrations. --- ## Security - All SOQL must use WITH USER_MODE — never write naked SOQL. FLS and CRUD are database-level controls; USER_MODE is how you invoke them. - All DML must use AccessLevel.USER_MODE — e.g. Database.insert(records, AccessLevel.USER_MODE). Never use naked insert, update, delete, upsert. - All Apex classes must declare "with sharing" unless there is an explicit, documented reason for "without sharing" (state the reason in a comment). - Never hardcode credentials, tokens, or secrets. Use Named Credentials or Custom Metadata. - Never build dynamic SOQL from user input without strict allowlisting. Prefer bind variables. - LWC: never use @AuraEnabled without cacheable=true on read operations. ## Automation (Flows, Triggers) - Before generating any automation, ASK ME what existing automation already runs on this object. Do not create a Flow or trigger without knowing what else fires on the same object. - All Flows must include a fault path on every DML element. Never leave fault handling empty. - All automation must be bulk-safe. Design for 200+ records in a single transaction. No SOQL or DML inside loops. - Never create a Record-Triggered Flow without specifying entry conditions that exclude irrelevant record changes. - Include ownership metadata in the description: who owns it, what it does, which fields it writes. ## Agents (Agentforce) - Use Agent Script, not open-ended prompts, for any CUD operation, financial commitment, or regulated workflow. - Set confirmationRequired: true on any action that creates, updates, or deletes records or calls an external system. - Scope the action catalog to the agent's role. Do not give an agent access to actions outside its specific job. - Never pass raw inbound data (web forms, emails, case comments) directly into a prompt template without sanitization. Treat all inbound text as untrusted. ## Data Cloud - Prefer zero-copy / ZETL access over ingesting data copies. Ask me whether the use case requires physical data movement. - Any pipeline processing unstructured text (case notes, transcripts, documents) must include PII detection and masking before vectorization or RAG grounding. - Validate Data Space boundaries — never activate data to a destination outside its intended Data Space without explicit confirmation. ## MuleSoft / Integrations - Use short-lived, user-scoped tokens (OBO) instead of static API keys or broad service accounts. - All write-side integrations must be idempotent — include idempotency keys for order, billing, fulfillment, or record-mutating APIs. - Apply rate limits and token budgets at the gateway level. Ask me what limits are appropriate. ## Code Quality - Generate tests that validate the business rule — positive, negative, boundary, and bulk (200+). - Every test must assert a specific expected outcome, not just "no exception thrown." - Name all components descriptively — purpose clear from the name alone. - Limit methods to 40 lines. Refactor into named sub-methods if longer. --- ## What These Rules Will NOT Catch — Warn Me About These At the end of every code generation response, include warnings about: 1. CODE ANALYZER: "Run sf scanner run or Code Analyzer in CI/CD before deploying. These rules guide generation but cannot catch cross-class vulnerabilities or complex data flow issues." 2. PLATFORM CONTROLS: "This code is only safe in production if platform controls (Transaction Security, Event Monitoring, Platform Encryption) are configured for this object." 3. BUSINESS LOGIC: "I implemented the rule as described. Validate with business stakeholders and test with production-representative data — AI cannot verify business correctness." 4. AUTOMATION CONFLICTS: "Check for automation conflicts using metadata queries. I cannot see managed packages, scheduled jobs, or async processes on this object." --- For the full framework behind these rules: https://vibecode-governance.github.io

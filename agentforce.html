Agentforce

Deterministic agent execution with observable reasoning paths

Agentforce combines generative conversation, topic classification, action selection, Flow/Apex invocation, knowledge retrieval, and Agent Script into an execution layer that can act on Salesforce data. The main concern is not just what the agent says, but what the agent decides to do: which topic it selects, which action it invokes, what data it retrieves, and whether the execution path is deterministic, auditable, and bounded.

Key Considerations

Secure Agentforce practices must cover topic design, prompt templates, Agent Script, action catalog scope, confirmationRequired, least-privilege permissions, Trace Panel observability, escalation logic, and deterministic handling of regulated or high-impact workflows.

Example Guardrails

Agent Script for Deterministic Logic

Use Agent Script, not open-ended prompts, for business rules, routing, escalation thresholds, CUD operations, regulated workflows, financial commitments, and action chaining. Agent Script provides deterministic execution paths that are auditable and predictable — the agent follows a defined script rather than reasoning its way through high-stakes decisions.

Role-Scoped Agent Actions

Give each agent only the Flow, Apex, prompt, and external actions required for its assigned role. A sales coach, service agent, or campaign agent should not see a broad action catalog that lets it update records, call systems, or trigger workflows outside its intended job. Actions not in the catalog physically cannot execute, regardless of what the reasoning layer produces.

Trace Panel + Runtime Observability

Enable Trace Panel and route agent telemetry to audit/logging pipelines so topic classification, condition evaluation, action selection, data access, and handoff decisions are attributable. When a stakeholder asks "why did the agent do this?" you have a complete, structured answer — not a black box.

The platform constrains execution regardless of what the reasoning produces. Even if the LLM's reasoning is compromised, it can only execute actions in its catalog, with its scoped permissions, through the Trust Layer. Governance lives in the execution layer, not the reasoning layer.

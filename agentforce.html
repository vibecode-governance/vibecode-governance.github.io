Agentforce

Deterministic execution with observable reasoning and bounded autonomy

Agentforce is where the governance model becomes most tangible. It's not a black-box AI that makes opaque decisions — it's a structured execution framework where every agent has defined topics, explicit actions, observable reasoning, and platform-enforced boundaries. Governance isn't bolted on afterward. It's the architecture.

Most conversations about "AI agents" treat them as unpredictable black boxes that need to be constrained. Agentforce inverts this. The agent's behavior is defined through a structured framework: topics define scope, actions define capability, guardrails define boundaries, and the Trust Layer provides the observation and enforcement layer. The result is an agent system where governance is a design choice, not an afterthought.

The difference between Agentforce and a generic LLM-powered agent: the platform constrains execution regardless of what the reasoning produces. Even if the LLM's reasoning is perfect, it can only execute actions in its catalog. Even if its reasoning is compromised, it still can't exceed its defined boundaries. Governance lives in the execution layer, not the reasoning layer.

The Governance Architecture

Agentforce agents are built from composable, governed primitives. Each layer adds a governance control:

Topics & Scope

Every agent has explicitly defined topics that determine what it can reason about. A "Case Resolution" agent has topics related to troubleshooting and resolution — not pricing, not HR, not financial data. If a user (or injected instruction) asks the agent to operate outside its topics, it recognizes the boundary and declines.

This is scope governance: the agent knows what it's for, and it won't be repurposed through conversational manipulation.

Action Catalog

Each topic maps to a defined set of actions the agent can execute. Actions are explicitly registered — they're not arbitrary API calls. The agent can only do what's in its catalog, and each action has defined inputs, outputs, and permission requirements.

This is capability governance: regardless of what the reasoning layer produces, execution is limited to the registered action set.

Guardrails & Instructions

System-level instructions define behavioral boundaries: what the agent must always do, what it must never do, and under what conditions it must escalate. These are set at the agent definition level — not in the conversation — so they can't be overridden by user input.

This is behavioral governance: the rules are in the system prompt, not the user prompt, and they persist across all interactions.

Trust Layer

Every LLM interaction routes through the Trust Layer — a centralized gateway that provides injection defense, toxicity detection, PII masking, and a complete audit trail. The agent doesn't access the LLM directly. The Trust Layer mediates every interaction.

This is interaction governance: every reasoning step is screened, logged, and defensible.

Bounded Autonomy in Practice

The term "bounded autonomy" means the agent can act independently within defined limits, and must escalate beyond those limits. Agentforce implements this through three mechanisms:

confirmationRequired

Mark high-risk actions so they require explicit user approval before execution. The agent recommends; the human confirms.

Action Catalogs

Whitelist what each agent role can do. Actions not in the catalog physically cannot execute, regardless of reasoning.

Escalation Triggers

Define conditions that force human handoff: confidence below threshold, topic outside scope, financial impact above limit.

Why this matters for governance: On generic LLM agent frameworks, the boundary between "what the agent can reason about" and "what the agent can do" is often blurred. Agentforce makes this separation architectural. The reasoning layer can consider anything. The execution layer only permits what's cataloged. This is how you get the benefit of flexible AI reasoning with the safety of deterministic execution.

Agentforce Governance Controls

Observable Reasoning

Every agent interaction produces an observable trace of the reasoning steps, actions considered, actions taken, and data accessed. This isn't just logging — it's structured observability that answers:

Why did the agent do this? The reasoning trace shows what information the agent considered and how it arrived at its decision.

What data did it access? Every query the agent executed is logged with the specific records and fields retrieved.

What did it not do? Actions considered but not taken (because of guardrails, confidence thresholds, or scope boundaries) are also visible in the trace.

This observability is what makes agent behavior auditable. When a stakeholder asks "why did the agent send this response?" you have a complete, structured answer.

Permission-Scoped Execution

Agentforce agents execute with defined permission sets — not as system administrators. This means:

Minimum necessary access. A "Lead Qualification" agent gets access to Leads and relevant custom objects. It doesn't get access to Opportunities, Contracts, or Financial data it doesn't need.

Sharing model enforcement. Agent queries respect the same sharing rules as any other user. An agent running on behalf of a specific user can only access records that user can access.

Field-level security. Even within accessible objects, the agent only sees fields its permission set grants access to. Sensitive fields can be excluded from agent access without removing them from the object entirely.

Testing & Deployment Governance

Agents go through a governed deployment lifecycle:

Sandbox testing with defined scenarios. Before production deployment, agents are tested against a suite of expected interactions, edge cases, and (critically) adversarial inputs. Does the agent stay within its topic when pushed? Does it escalate when it should?

Graduated rollout. Agents deploy to a subset of users first. Performance metrics (resolution accuracy, escalation rates, user satisfaction) are evaluated before full rollout.

Continuous monitoring post-deployment. Event Monitoring tracks agent behavior patterns. Anomalies (unusual action frequency, unexpected data access, high escalation rates) trigger review.

The Agentforce principle: Governance isn't a tax on agent capability — it's what makes autonomous agents deployable in production. Without bounded autonomy, observable reasoning, and permission-scoped execution, you can't put an agent in front of customers or let it touch revenue data. With them, you can. That's the business value of governance-by-architecture.

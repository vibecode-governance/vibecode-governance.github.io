Failure Mode #4: State Corruption & Automation Sprawl

Flow Sprawl: Sandbox Success, Production Failure

AI generates Flows that work in isolation but produce cascading side effects at scale

A good CI/CD pipeline catches the obvious problems — naming violations, missing fault paths, bulk-unsafe patterns. What it doesn't catch is semantic interaction: Flows that individually pass every gate but produce cascading side effects when they execute together on the same record at scale.

Bad Prompt

"Create a Salesforce Flow that updates Opportunities when stage changes and sends a notification to sales."

Likely Bad Output

The AI generates a Flow that:

— Has too many elements and unnecessary loops

— Is not bulk-safe (works for 1 record, crashes at 200+)

— Lacks fault paths (failures are silent)

— Has no awareness of other automation already on the same object

— Gets a generic name with no description or ownership metadata

— Does not account for FLS or sharing

What Goes Wrong

The Flow works fine in sandbox with one test record. In production, three developers have independently generated three Flows on the same object. Each passes its own tests. Each passes CI/CD individually. But when a batch operation fires all three on the same records:

Flow B reassigns the owner → that field change re-triggers Flow A → close dates get recalculated → Flow C fires a notification on every cascading update. Result: wrong data on hundreds of records, thousands of spurious notifications, and hours to trace which Flow caused what.

No individual test caught it because no test ran them together on the same batch. The pipeline saw no structural overlap because they trigger on different field conditions. The conflict is emergent, not structural — it only surfaces at runtime, at scale.

Good Prompt

"Generate a production-safe Salesforce Flow design for Opportunity stage change handling. Keep it bulk-safe, minimize elements, avoid unnecessary loops, include fault paths, assume enterprise scale. There is already a Record-Triggered Flow on Opportunity that reassigns owner when Amount exceeds $500K — your new automation must not conflict with this existing behavior. Provide recommended naming, ownership metadata, and deployment controls."

Better Output

The AI now generates a Flow that:

— Is simplified with minimal element count

— Includes fault paths that route to error handling

— Is bulk-safe (no DML/SOQL inside loops)

— Explicitly avoids conflict with the existing owner-reassignment Flow

— Follows naming convention: Opportunity_StageChangeNotify_v1

— Includes description with owner, requirement reference, and fields affected

Prevention Pattern

Constrain generation with context, then validate in pipeline.

CI/CD catches structural problems. The prompt itself must prevent semantic conflicts — because cascading side effects between Flows are invisible to static analysis.

// What CI/CD catches (structural)
✓ Naming convention enforced
✓ Description populated with owner + requirement
✓ Fault paths present
✓ No DML/SOQL inside loops
✓ Basic overlap detection (same object + same trigger)

// What the prompt must prevent (semantic)
→ Cascading field updates that re-trigger other Flows
→ Side effects that interact at scale but not in single-record tests
→ Automation that duplicates or contradicts existing logic

Best Practices

Include existing automation context in the prompt. Tell the AI what's already deployed on the target object before it generates anything new.

Prompt for bulk-safe design and minimal elements. Explicitly request fault paths and no unnecessary loops.

Require naming and ownership metadata. Every Flow needs a standardized name, description, owner, and requirement reference.

Use Flow Trigger Explorer during design review. Check all automation on the object in execution order before building.

Monitor with Event Monitoring post-deployment. Watch for execution spikes that indicate cascading interactions at scale.

Use Field Audit Trail for forensics. When interactions do occur, trace exactly which process changed what.

CI/CD catches structural problems. Good prompts prevent the interaction problems CI/CD can't see. The governance layer here is the context you feed the AI before generation — not just the pipeline that validates after.

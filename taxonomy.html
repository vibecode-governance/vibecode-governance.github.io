Vibe Coding Failure Taxonomy

What actually goes wrong in AI-assisted Salesforce development

Failures in the agentic SDLC are rarely syntax errors. They are semantic — the code compiles, the tests pass, and the business logic is wrong. Or the security model is silently bypassed. Or the agent treats customer text as an instruction.

Before you can govern vibe coding effectively, you need a clear picture of what actually breaks. Not theoretical risks — practical failure modes that show up in Salesforce development when AI generates code and agents act on it.

This taxonomy covers four technical failure vectors plus one organizational failure mode. Each one maps to specific controls in the Governance Model and is illustrated with a concrete before/after example in the demos section.

Once agents can act, not just suggest, every gap in governance becomes an execution path. The taxonomy below isn't academic — each failure mode represents a way that AI-generated or AI-executed logic can affect production data, customer experience, or security posture.

01

Logic Hallucination

The code works. The business rule is wrong.

This is the most insidious failure mode because everything looks correct. The code compiles. It may even have test coverage. But the underlying business logic doesn't match the actual requirement. The AI optimized for plausibility, not accuracy.

Classic patterns include: discounting logic that applies to all orders instead of only those above a threshold, boundary conditions handled backward, agents that optimize for a "reasonable-sounding" response over a factually correct one (sometimes called "LLM sycophancy" or "lies-in-the-loop").

Salesforce example A developer asks the AI to "apply a 10% discount for orders above $100." The AI generates a method that applies the discount to every order — no threshold check. The code is clean, compiles perfectly, and even passes a basic test (because the test only checks orders above $100). The boundary condition at exactly $100 is never validated. Money leaks quietly.
Why traditional review misses it The code looks like correct Apex. There's no syntax error, no security warning, no governor limit violation. A human reviewer scanning for "does this look right" will often miss it — especially under the volume pressure of AI-generated PRs. Only spec-first development with explicit boundary assertions catches this reliably.

Detection requires:

Spec-driven development Test-first prompting Boundary assertions Negative test cases

Prevention pattern: Write the tests first. When the AI writes tests before implementation, it's forced to align to the rule. When it writes code first, it optimizes for plausibility. The order matters.

02

Security Bypass

AI defaults to system mode — platform security never invoked.

AI-generated code optimizes for compilation and functional correctness. It does not optimize for the bipartite security model of the Salesforce platform. The result: code that runs in system mode by default and never explicitly invokes user-context enforcement — so FLS, CRUD, and sharing rules (which are database-level controls) are simply never applied to that transaction.

The most common patterns on Salesforce: missing with sharing, naked SOQL (without WITH USER_MODE), naked DML (without AccessLevel.USER_MODE), deprecated WITH SECURITY_ENFORCED usage, and hardcoded credentials in integration classes. On general platforms: missing authentication, SSRF from hallucinated URLs, unvalidated redirects.

Salesforce example A developer asks the AI to create an Apex method that updates Account revenue and creates a Task. The AI generates clean code that compiles and works. But it uses update acc; instead of Database.update(acc, AccessLevel.USER_MODE), runs SOQL without WITH USER_MODE, and omits with sharing on the class. In sandbox with an admin user, it works perfectly. In production, it runs in system mode — the platform's access controls exist but the code never invokes them, so every user gets admin-equivalent data access.
Why this is a platform-specific risk Salesforce's security model is two-layered: record-level (sharing rules, role hierarchy, ownership) and attribute-level (CRUD on objects, FLS on fields). Generic AI models don't understand this architecture — they've seen thousands of examples of Apex written in system mode because that's what most public code looks like. The path of least resistance produces working-but-unsafe code every time.

Detection requires:

Code Analyzer + DFA in CI/CD WITH USER_MODE enforcement Naked SOQL/DML scanning with sharing policy

Prevention pattern: Enforce security at the database boundary. Don't rely on the AI to "remember" security — push security enforcement into the query and DML layer. Make the secure pattern the default prompt pattern so the AI generates it automatically.

03

Prompt Injection & Agent Hijack

Data becomes instruction. The agent is co-opted.

This is the failure mode that's unique to the agentic era. When an agent reads untrusted text — from a case description, a lead form, a web chat, a PDF — it may treat embedded instructions as commands rather than content. The agent's reasoning is hijacked without any authentication bypass, code exploit, or network intrusion.

Most dangerous when the agent sits in what researchers call the "lethal trifecta": (1) access to untrusted input, (2) access to private data, and (3) ability to produce external output. This is the ForcedLeak class of exploit.

Salesforce example — the ForcedLeak chain An external attacker submits a Web-to-Lead form. The Description field (up to ~42,000 characters) contains embedded LLM instructions disguised as prose. The poisoned Lead sits in the org, unread. Later, an internal sales rep asks Agentforce to "summarize my newly assigned leads." The agent retrieves the poisoned Lead, treats the embedded instructions as commands, queries Opportunity data, and renders the results in a markdown image URL. The browser loads the "image" — and the query string contains the exfiltrated data. No code was executed. No authentication was needed. The rep never clicked anything suspicious.
What makes this different from traditional security vulnerabilities The attacker never authenticates. Never touches the Salesforce UI. Never runs code in the org. The internal user is the unwitting courier. Traditional perimeter defenses don't see this — it looks like normal agent activity from the inside.

Defense requires layers (no single control is sufficient):

Trust Layer injection defense Trusted URLs + strict CSP (img-src) confirmationRequired on CUD actions Least-privilege agent permissions Bounded autonomy with forced escalation

Prevention pattern: Separate data from execution and bound the agent. Treat all CRM text fields as untrusted input. Restrict outbound image/connect sources to explicit allowlists. Require confirmation for high-risk actions. Limit the agent's action catalog to only what it needs for its assigned role.

04

State Corruption & Automation Sprawl

Multiple AI-generated automations on the same object — nobody knows which one is changing the data.

AI generates a Flow or trigger that works perfectly for one record in sandbox. But production has existing automation on the same object, bulk data loads that fire it thousands of times, and no one tracking which automation owns which field. The result: conflicting automations silently corrupt data and you don't find out until a report doesn't add up weeks later.

Salesforce example A developer asks the AI to create a Flow that updates Opportunities on stage change and notifies sales. It works for one record in sandbox. In production, a data migration touches 5,000 records — the Flow fires 5,000 times, conflicts with an existing trigger on the same object, creates duplicate tasks, and corrupts stage history. The pipeline report breaks three weeks later.
Related patterns Three automations on the same object with no one knowing which sets a field value. Duplicate records from AI-generated transforms. Conflicting identity resolution rules in Data Cloud. Schema bloat from AI-hallucinated redundant fields.

Detection requires:

Automation overlap detection Bulk-safe design enforcement Metadata validation in CI/CD Naming and ownership standards

Prevention pattern: Before generating any automation, map what already exists on that object. The prompt must include existing triggers, Flows, and process builders. CI/CD must validate that no two automations write to the same field without explicit conflict resolution.

05

Comprehension Debt

The team can't debug what the AI wrote. A people problem, not a code problem.

This isn't a technical bug — it's an organizational risk. When developers approve AI-generated code faster than they can reason about it, the organization builds a system more sophisticated than any individual can debug. The AI generated it in seconds. Understanding it takes hours. And when it breaks at 3 AM, nobody has that mental model.

Comprehension debt is distinct from technical debt. Technical debt is code you understand but chose not to fix yet. Comprehension debt is code that's already in production that no one fully understands. It's harder to pay down because you don't even know where the balance is.

How it accumulates A team of five developers uses AI assistants across 20 PRs per week. Each PR contains code that was generated in seconds and reviewed in minutes. The review catches obvious errors but doesn't build a deep mental model of what the code does or why it's structured the way it is. After six months, the codebase contains thousands of lines that were never fully understood by anyone currently on the team. When a production incident requires tracing through this code, resolution time explodes.
A sub-pattern: "Pattern Invention" LLMs sometimes generate code that calls APIs, libraries, or framework-specific methods that don't actually exist — they're hallucinated from plausible-sounding patterns in the training data. On Salesforce this might look like a call to a non-existent Apex method, a fabricated governor limit workaround, or reference to an LWC API that doesn't exist in the current version. If the developer doesn't catch it (and remember — 56% don't review line by line), it either fails in production or creates confusing behavior that's extremely hard to debug.

Mitigation requires organizational practices:

Knowledge transfer rituals (explain the AI code in the PR) Spec-driven development (human spec is source of truth) AI Code Review Standard (5-question checklist) Periodic "manual sprints" to prevent skill atrophy Code Turnover metric tracking

Prevention pattern: The developer's primary cognitive load must shift from generating code to understanding, validating, and governing it. This requires organizational change — updated review standards, new metrics, and career incentives that reward comprehension over velocity.

How These Failure Modes Interact

In practice, these failure modes rarely appear in isolation. A typical incident might combine multiple vectors:

Logic hallucination + security bypass: AI generates a discount method that applies to all orders (logic hallucination) and also uses naked DML without access checks (security bypass). The wrong discount gets applied to records the user shouldn't even be able to update.

Prompt injection + state corruption: An agent is hijacked into updating records based on injected instructions. The updates are valid from a permissions standpoint but semantically wrong. The data is corrupted in a way that looks like normal agent activity.

Automation sprawl + comprehension debt: AI generates multiple Flows over time, each solving a narrow problem. They interact in ways nobody planned for. When bulk data processing triggers all of them simultaneously, the interactions create state corruption that's almost impossible to trace because no one understands the full picture.

This is why the governance model has five pillars, not one. A linter catches security bypass. Spec-first development catches logic hallucination. The Trust Layer catches prompt injection. Metadata validation catches automation sprawl. Organizational practices address comprehension debt. No single control covers all five failure modes.

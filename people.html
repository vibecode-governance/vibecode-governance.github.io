People & Organization

Roles, metrics, interviews, and the concept of "taste" in AI-assisted development

Platform controls catch the technical failures. People controls address the organizational ones.

When developers can generate code faster than they can understand it, and when traditional metrics reward volume over value, the governance model needs a human layer that's intentionally designed for the AI-assisted world.

This is Pillar 5 of the governance model — and in many ways the hardest to implement because it requires changing how we measure, hire, promote, and organize development teams. The technical pillars require configuration. The people pillar requires culture.

The developer's role is evolving from "person who writes code" to "person who validates AI output against business intent." This isn't a reduction in skill — it's a shift in the cognitive load from syntax to judgment, from generation to evaluation, from typing to thinking. Organizations that recognize and reward this shift will govern AI-assisted development effectively. Those that don't will accumulate comprehension debt until it becomes unmanageable.

The Evolving Career Path

In the AI-assisted era, seniority is no longer defined by how much code you can write. It's defined by how much complexity you can govern. Here's what the career ladder looks like:

Junior Developer → Curator

Core responsibility: Evaluate AI output. Can the developer look at generated code and determine if it's correct, secure, and aligned with the requirement?

Key skills: Reading code critically. Identifying boundary conditions the AI missed. Running and interpreting test results. Flagging code they don't understand (rather than approving it).

Growth signal: They catch errors that the AI generated confidently. They ask "what happens at the boundary?" before approving. They can explain WHY the code works, not just that it works.

Mid-Level Developer → Pipeline Owner

Core responsibility: Design the constraints that shape AI output. Write the specs. Define the test boundaries. Own the prompt templates and review standards for their area.

Key skills: Spec-driven development. Designing prompt constraints that produce secure code by default. Building and maintaining CI/CD validation rules. Mentoring juniors on evaluation techniques.

Growth signal: The AI generates better code in their area because their constraints are well-defined. Fewer issues escape to production from their team. They can articulate why a particular AI output is wrong at an architectural level, not just a syntactic one.

Senior Developer → Constraint Architect

Core responsibility: Define organizational policy for AI-assisted development. Set the governance standards. Design the automation registries. Establish the review protocols that scale across teams.

Key skills: System-level thinking. Understanding how multiple AI-generated components interact. Designing governance frameworks that enable velocity without sacrificing safety. Making the secure path the easy path.

Growth signal: The organization produces better AI-assisted code because of frameworks they put in place. They can look at a codebase and identify comprehension debt before it becomes an incident. They define what "taste" means for their organization.

The Concept of "Taste"

In a world where AI can generate any plausible-looking code, the critical skill is rejection — the ability to look at functionally correct output and say "this is wrong for our architecture, our scale, our security model." We call this skill "taste."

Taste is what separates a developer who approves AI output from one who governs it. It's the judgment to recognize:

"This works but won't scale." The AI generated a solution that handles 10 records. Your production environment processes 100,000. The code is correct but architecturally inappropriate.

"This is secure but unmaintainable." The AI generated a complex, deeply nested method that's technically safe but that nobody on the team can debug when it breaks at 3 AM.

"This passes tests but violates our patterns." The AI generated an implementation that works but contradicts the architectural decisions the team has already made. Accepting it creates inconsistency that compounds over time.

"This looks right but the business rule is wrong." The most subtle form of taste — recognizing that plausible code implements the wrong intent. This requires domain knowledge that no linter can replicate.

How to develop taste: It doesn't come from reading documentation. It comes from experience with production systems, exposure to incidents, and deliberate practice evaluating AI output. Organizations that want to build taste need to create the conditions for it: code review rituals that reward critical evaluation, periodic "manual sprints" where teams implement without AI to maintain hands-on skills, and post-incident reviews that trace failures back to governance gaps.

Metrics That Matter in the AI Era

Traditional development metrics (lines of code, PRs merged, velocity points) reward volume. In AI-assisted development, volume is trivially achievable. The metrics that matter now measure value, quality, and sustainability:

TrueThroughput
Value delivered, not volume produced

Features that reach production and stay there (don't get reverted). Measures actual business value rather than raw output.

Code Turnover
< 12% (30-day)

Percentage of code modified or deleted within 30 days of creation. High turnover indicates the team is generating code faster than they can get it right.

Maintainability Index
> 70

Composite score measuring code complexity, documentation, and test coverage. Ensures AI-generated code is understandable and maintainable by the team.

PR Cycle Time
< 8 hours

Time from PR creation to merge. A balance point: too fast may indicate rubber-stamping; too slow indicates review bottleneck.

Spec Coverage
100% for critical paths

Percentage of AI-generated code that has a corresponding human-written spec. No spec = no governance over what the AI produces.

Cost Realization
Positive ROI within quarter

Are AI tools actually saving time when you account for rework, debugging, and incident response? The velocity gain only counts if it sticks.

The AI Code Review Standard

Every PR that contains AI-generated code should receive a structured review against five questions. This isn't about slowing things down — it's about making review intentional rather than cursory:

  1. Does it match the spec? Compare the implementation against the human-written requirement. Not "does it look right" but "does it do what we specified, including edge cases?"
  2. Does it handle boundaries correctly? Test the threshold conditions, the zero cases, the null cases, the maximum cases. These are where logic hallucinations hide.
  3. Is it secure? with sharing present? USER_MODE on queries and DML? No hardcoded credentials? No naked SOQL/DML? Code Analyzer clean?
  4. Is it bulk-safe? No DML in loops? No queries in loops? Handles 200+ records without hitting governor limits? Has fault paths?
  5. Can the team maintain it? Can another developer on the team explain what this code does and why? If not, it's adding comprehension debt regardless of whether it works.

The standard isn't meant to be a gate that blocks everything. It's meant to focus reviewer attention on the five areas where AI-generated code most commonly fails. Questions 1 and 2 catch logic hallucination. Question 3 catches security bypass. Question 4 catches state corruption. Question 5 addresses comprehension debt.

Hiring for the AI-Assisted World

Traditional coding interviews test whether a candidate can write code from scratch under time pressure. This is no longer the primary skill. The interview process should evaluate what matters now: can they govern AI output effectively?

Stage 1: Constraint Definition

Give the candidate a business requirement. Ask them to write the spec — not the code. Can they define inputs, outputs, boundary conditions, error states, and security requirements clearly enough that an AI would generate correct output?

What you're evaluating: Do they think about edge cases before implementation? Do they consider security as part of the spec? Do they define what "correct" means before asking for code?

Stage 2: Debugging Hallucinations

Show the candidate AI-generated code that contains subtle errors (logic hallucination, security bypass, bulk-safety issue). Can they identify what's wrong? Can they explain why it's wrong?

What you're evaluating: Can they read code critically rather than assuming it's correct? Do they check boundaries? Do they notice missing security patterns? Do they question plausible-looking output?

Stage 3: Demonstrating Taste

Present the candidate with multiple AI-generated solutions to the same problem. All are functionally correct. Ask them which they'd choose and why. Can they articulate architectural reasoning beyond "it works"?

What you're evaluating: Do they consider maintainability, scale, team comprehension, and consistency with existing patterns? Can they reject "correct" code for legitimate architectural reasons? This is taste.

Practical Organizational Controls

Spec-before-generate policy. No AI code generation without a written spec. This can be lightweight (a structured comment block with inputs/outputs/boundaries) but it must exist. The spec is the governance artifact.

PR attribution. Every PR clearly identifies which portions are AI-generated. This isn't about blame — it's about directing reviewer attention. AI-generated code gets the five-question review. Human-written code gets standard review.

Periodic manual sprints. Once a quarter, the team implements a feature without AI assistance. This maintains hands-on skills, prevents tool dependency, and builds the muscle memory that makes "taste" possible.

Comprehension debt tracking. Identify code in production that nobody on the current team fully understands. Treat it like any other form of tech debt: track it, prioritize it, and allocate time to pay it down.

Knowledge transfer in PRs. AI-generated PRs include a brief explanation (in the PR description, not just in code comments) of what the code does and why it's structured this way. This forces the author to understand what they're shipping.

The organizational principle: Technology controls prevent the failure modes. People controls prevent the conditions that create failure modes. If your organization rewards velocity without measuring quality, if interviews test coding speed rather than governance judgment, if career paths don't value "taste" — no amount of platform control will prevent comprehension debt from accumulating.

Interview Question Bank

Example questions for evaluating AI-assisted development skills at different levels. These are organized by general development role and by Salesforce-specific expertise.

General Development — Junior / Curator
  1. Here's a piece of AI-generated code. Walk me through what it does line by line. Where would you want to see tests?
  2. This function works for 1 record. What happens if 200 records come in at once? What would you check?
  3. The AI generated this code and all tests pass. What would make you hesitate before approving it?
  4. You asked the AI to build a login form. The code works. What security issues would you look for that the AI might have missed?
  5. Describe a situation where code that compiles and passes tests is still wrong. How would you catch it?
  6. You don't understand a section of code the AI generated. What do you do? (Looking for: "I flag it and ask" vs. "I approve it anyway.")
  7. The AI generated three possible solutions. All work. How do you decide which one to use?
General Development — Mid-Level / Pipeline Owner
  1. How would you write a spec for a feature such that AI generates correct, secure code on the first try? Walk me through your process.
  2. You're reviewing a PR from a junior who accepted AI-generated code. What's your review checklist? What specifically do you look for?
  3. Two different AI-generated automations modify the same data. How do you design a system that prevents this conflict before it reaches production?
  4. You notice that AI-generated code in your team's area has a high revert rate. How do you diagnose why, and what do you change?
  5. How would you set up a CI/CD gate that catches AI-specific failure patterns (security bypass, bulk-safety, missing error handling) without slowing deployment to a crawl?
  6. A junior on your team is approving AI output without really understanding it. How do you address this without making them feel micromanaged?
  7. Describe how you'd design prompt constraints or templates so that AI generates code that follows your team's patterns by default.
General Development — Senior / Constraint Architect
  1. Your organization just adopted AI coding tools across 50 developers. What governance framework do you put in place in the first 90 days?
  2. How do you measure whether AI-assisted development is actually making the team more effective — vs. just faster at producing code that creates rework downstream?
  3. You have a codebase where half the code was AI-generated six months ago and the original developers have left. How do you assess and manage the comprehension debt?
  4. Design an automation registry for an org with 15 teams all using AI to generate Flows. How do you prevent sprawl and conflicts at organizational scale?
  5. What's the difference between code that's functionally correct and code that demonstrates architectural taste? Give me a concrete example.
  6. A CIO asks you: "How do I know my AI-generated code is safe?" What's your answer — not the tools, but the organizational system you'd design?
  7. How do you evolve a career ladder and performance reviews when traditional metrics (lines of code, PRs merged, velocity points) no longer measure what matters?
Salesforce-Specific — Developer / Admin
  1. AI generated an Apex class without with sharing. What's the risk? When would without sharing actually be appropriate?
  2. Explain the difference between WITH USER_MODE and WITH SECURITY_ENFORCED. Which would you use and why?
  3. The AI generated a Record-Triggered Flow that works in sandbox. What three things would you check before deploying to production?
  4. You asked the AI to write a trigger. It has SOQL inside a for loop. Why is this a problem on Salesforce specifically, and how do you fix it?
  5. An AI-generated Flow has no fault path on its DML elements. What happens in production when one of those DML operations fails?
  6. The AI generated a @future method to call an external API. What governor limit concerns would you raise? What alternative patterns exist?
  7. You're told there are already two triggers and a Flow on the Account object. The AI is asked to create another Flow on Account. What questions do you ask before allowing it?
  8. How does Platform Encryption affect AI-generated SOQL queries? What can and can't you do with encrypted fields?
Salesforce-Specific — Architect / Expert
  1. Walk me through how you'd design a governance model for an org where 20 citizen developers are using AI to generate Flows. What controls exist natively on the platform?
  2. An Agentforce agent has access to a broad action catalog. How do you scope it down, and what's the risk if you don't? Explain confirmationRequired and Agent Script in this context.
  3. Describe the Salesforce Trust Layer. How does it intercept prompt injection? What happens if an inbound Web-to-Lead form contains malicious instructions?
  4. A developer used AI to generate a Data Cloud ingestion pipeline. What governance checks should exist before it reaches production? Think about PII, identity resolution, and Data Spaces.
  5. How do Transaction Security Policies and Event Monitoring work together to provide runtime governance? Give an example of a policy that would block a data exfiltration attempt in real time.
  6. You're designing a shared responsibility model between the platform team and application teams. What does the platform enforce automatically, and what must each team configure?
  7. A MuleSoft integration is being invoked by an Agentforce agent. Walk me through the security chain: how does OBO work, what's the blast radius if the agent is compromised, and where are the control points?
  8. How would you use Security Center, Shield, and Code Analyzer together to create end-to-end governance across a multi-cloud Salesforce estate with AI-generated code?

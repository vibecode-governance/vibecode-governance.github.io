Platform as Governance Layer

How Salesforce primitives map to every control in the framework

This is the thesis of the entire framework: Salesforce platform products — especially Shield — function as a governance layer for AI-assisted development. The controls aren't aftermarket add-ons. They're built into the platform architecture. Every governance requirement in this framework maps to a product capability that already exists.

Throughout this framework, we've referenced specific Salesforce capabilities: Trust Layer, Platform Encryption, Event Monitoring, Transaction Security, Code Analyzer, USER_MODE enforcement. This page brings them all together in one reference — showing exactly how the platform's existing architecture addresses each pillar and each failure mode.

Salesforce secures the platform. You secure what you build on it. The shared responsibility model is clear: infrastructure security, data isolation, Trust Layer operation, and platform encryption are Salesforce's responsibility. Governance of custom code, agent configurations, automation logic, and organizational practices is yours. This framework tells you exactly how to fulfill your side — using the tools the platform provides.

Complete Control-to-Product Mapping

Shield: The Core Governance Suite

Platform Encryption

Governance role: Protects sensitive data at rest with tenant-managed keys. Even if AI-generated code bypasses access controls at the application layer, the field values are encrypted at the database level.

AI-era application: Encrypt revenue fields, competitive intelligence, customer PII, and any data that agents access for grounding. Provides a safety net below the application layer.

Addresses: Security Bypass, Prompt Injection (limits damage of successful exfiltration)

Field Audit Trail

Governance role: Retains field-level change history for up to 10 years. Every modification is attributed to a user or process, with old value, new value, and timestamp.

AI-era application: When an agent modifies data, trace exactly what changed. When automation creates unexpected state, identify which process is responsible. Provides the forensic record for incident response.

Addresses: State Corruption, Logic Hallucination (post-incident tracing), Agent Hijack (attribution)

Event Monitoring

Governance role: Captures detailed event logs for user activity, API calls, report exports, login events, and more. Enables behavioral analysis and anomaly detection.

AI-era application: Monitor agent activity patterns. Detect when agents access data outside their normal scope. Surface bulk operations that may indicate runaway automation. Feed into Transaction Security Policies for real-time enforcement.

Addresses: All failure modes (detection and response layer)

Data Detect

Governance role: Automatically identifies and classifies sensitive data (PII, PCI, PHI) across the org. Scans existing data to find sensitive information in unexpected locations.

AI-era application: Before giving agents access to data, know where the sensitive data lives. Classify fields so that access policies can be applied correctly. Find PII that's ended up in free-text fields where agents might read it.

Addresses: Security Bypass (know what to protect), Prompt Injection (identify sensitive fields agents access)

Security Center

Governance role: Unified security posture dashboard across all orgs. Monitors compliance with security policies, surfaces configuration drift, and provides cross-org visibility.

AI-era application: As AI-assisted development accelerates across multiple orgs, Security Center ensures governance standards are applied consistently. Detect when one org drifts from the security baseline. Compare agent configurations across environments.

Addresses: Organizational governance and compliance visibility

Trust Layer & Runtime Controls

Trust Layer (LLM Gateway)

Governance role: Centralized gateway for all LLM interactions. Provides injection defense, toxicity detection, output quality scoring, and complete audit trail. No direct LLM access — everything routes through the Trust Layer.

AI-era application: The single most important runtime control for agent governance. Screens agent inputs for injection patterns. Screens agent outputs for harmful or toxic content. Scores response quality. Logs every interaction for compliance.

Addresses: Prompt Injection (primary defense), Security Bypass (output screening), all failure modes (audit trail)

Transaction Security Policies

Governance role: Real-time policy engine that evaluates events against defined conditions and can block, require MFA, or notify in real time. Fires before the action completes.

AI-era application: Block bulk data exports that match exfiltration patterns. Require elevated authentication when agents attempt high-risk operations. Alert on unusual agent behavior in real time rather than after the fact.

Addresses: Prompt Injection (real-time blocking), State Corruption (prevent bulk damage)

Development & Deployment Controls

Code Analyzer + Data Flow Analysis

Governance role: Static analysis tool that detects security anti-patterns in Apex code. Data Flow Analysis traces how data moves through the code to identify vulnerabilities that simple pattern-matching would miss.

AI-era application: Run in CI/CD to catch AI-generated code with missing security patterns (naked SOQL/DML, missing with sharing, hardcoded credentials) before it reaches production. Automated enforcement of security standards regardless of code origin.

Addresses: Security Bypass (primary build-time defense)

DevOps Center + Metadata API

Governance role: Structured deployment pipeline with change tracking, environment management, and deployment history.

AI-era application: Pipeline gates for AI-generated components. Metadata validation rules (naming, descriptions, ownership). Automation overlap detection before deployment. Complete audit trail of what was deployed, when, and by whom.

Addresses: State Corruption (prevent bad automation from deploying), Comprehension Debt (attribution and tracking)

The platform thesis, stated plainly: On most platforms, you build AI governance from scratch. On Salesforce, three out of five pillars already have platform-native controls. Your job is to activate them (Shield, Trust Layer, Code Analyzer) and build the two that require organizational investment (Design practices, People controls). The framework shows you exactly how to do both.

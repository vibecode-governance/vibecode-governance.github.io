Failure Mode #1: Logic Hallucination

Logic Hallucination: Looks Correct, Is Wrong

When AI-generated code compiles, passes tests, and implements the wrong business rule

A developer asks an AI assistant: "Apply a 10% loyalty discount for orders above $100." The code comes back clean, readable, well-structured. It applies a 10% discount to every order. The $100 threshold check is missing. Money leaks quietly.

The Scenario

This is the most insidious failure mode because there's nothing obviously wrong. No syntax error. No security warning. No governor limit violation. The code looks like correct Apex. A reviewer scanning for "does this look right" will often approve it — especially under the volume pressure of AI-generated PRs.

The problem isn't that the AI can't code. It's that the AI optimized for plausibility rather than accuracy. The generated code is the most likely-looking implementation of the prompt — but "most likely-looking" and "correct" are different things, especially at boundary conditions.

The Developer's Prompt

"Write an Apex method that applies a 10% loyalty discount to orders above $100. The method should take an Order record and return the discounted amount."

Sounds clear enough, right? Let's see what comes back.

What the AI Generates vs. What the Business Needs

What the AI generates
public class OrderDiscountService {

  public static Decimal applyLoyaltyDiscount(
    Order__c order
  ) {
    Decimal discountRate = 0.10;
    Decimal originalAmount = order.Total_Amount__c;

    // Apply 10% loyalty discount
    Decimal discountedAmount =
      originalAmount * (1 - discountRate);

    return discountedAmount;
    // Always applies discount — no threshold!
  }
}
What the business actually needs
public class OrderDiscountService {

  private static final Decimal DISCOUNT_RATE = 0.10;
  private static final Decimal MIN_THRESHOLD = 100.00;

  public static Decimal applyLoyaltyDiscount(
    Order__c order
  ) {
    Decimal originalAmount = order.Total_Amount__c;

    if (originalAmount > MIN_THRESHOLD) {
      return originalAmount * (1 - DISCOUNT_RATE);
    }

    // Below threshold — no discount
    return originalAmount;
  }
}

Why This Is Hard to Catch

Look at the left side again. It's clean code. Good variable names. A comment that describes what it does. If you're scanning PRs quickly — which everyone does when volume triples — this looks correct. The method name says "loyalty discount." The code applies a discount. Your brain fills in the threshold check because the prompt mentioned it.

Now look at what a typical test would look like:

@isTest
static void testLoyaltyDiscount() {
  Order__c order = new Order__c(Total_Amount__c = 150.00);
  Decimal result = OrderDiscountService.applyLoyaltyDiscount(order);
  System.assertEquals(135.00, result); // Passes!
}

The test passes. Of course it passes — $150 is above $100, so the discount should apply. The test only checks the happy path. It never tests the boundary: what happens at $99? What happens at exactly $100? What happens at $0?

This is the trap: When the AI writes both the code AND the test, both tend to validate the same plausible interpretation. The test confirms the code works — but it doesn't confirm the code is right. A test derived from the business spec would include boundary cases. A test derived from the code just mirrors its logic.

The Test Cases That Would Have Caught It

A spec-driven approach writes the test cases from the business requirement, not from the generated code:

Order = $150 — above threshold
Expected: $135
Both versions: $135 ✓
Order = $100 — at threshold (boundary)
Expected: $100 (not above)
AI version: $90 ✗
Order = $99.99 — below threshold
Expected: $99.99 (no discount)
AI version: $89.99 ✗
Order = $50 — well below
Expected: $50 (no discount)
AI version: $45 ✗
Order = $0 — zero edge case
Expected: $0
Both versions: $0 ✓

Three out of five test cases fail on the AI-generated version. But notice — the one test case the AI would naturally generate ($150) passes fine. This is why the order of operations matters: write tests from the spec first, then generate the implementation.

If this code processes 1,000 orders/month averaging $75 (below threshold)
$7,500/month in unintended discounts
$90,000/year in revenue leakage — from one missed boundary condition

The Governed Approach: Spec-First Prompting

The fix isn't "review more carefully." At AI-assisted velocity, you can't catch semantic errors through manual review alone. The fix is structural: write the spec (including boundaries and edge cases) before asking the AI to generate anything.

A governed prompt for the same requirement:

Write an Apex method with these specifications:

Method: applyLoyaltyDiscount(Order__c order) returns Decimal
Rule: Apply 10% discount ONLY when Total_Amount__c > 100.00
Rule: Orders at or below $100.00 receive NO discount
Rule: Return the final amount (not the discount value)

Boundary cases:
  - $100.00 exactly → return $100.00 (no discount)
  - $100.01 → return $90.01 (discount applies)
  - $0 → return $0
  - Null amount → throw IllegalArgumentException

Constants: Use named constants for threshold and rate
Tests: Write tests FIRST covering all boundary cases above

When you give the AI this prompt, it generates the correct implementation because the boundaries are explicit. The spec leaves no room for "plausible interpretation." And critically, the tests are derived from the spec — not from the generated code — so they actually validate correctness.

The spec is the governance artifact. In traditional development, the spec was documentation. In AI-assisted development, the spec is the constraint that prevents hallucination. If your team isn't writing specs before generating code, they're accepting whatever the AI finds most plausible — and plausible isn't the same as correct.

This Pattern Shows Up Everywhere

The discount example is simple and easy to spot. In production, logic hallucinations are far subtler:

CPQ pricing logic where a tiered discount applies to the wrong quantity brackets. Revenue recognition is off by small amounts per deal — invisible until quarter close.

Lead scoring models where the AI implements a reasonable-looking formula that weights the wrong fields or applies thresholds in the wrong direction. Marketing complains that "the scores don't match reality" but can't point to a specific bug.

Approval routing where the AI generates a Flow that routes to the correct approver for deals above $500K but silently skips approval for deals at exactly $500K. Those deals auto-approve without anyone noticing.

Date calculations where the AI handles "30 days from today" correctly in most months but fails on month boundaries, leap years, or time zone transitions. The errors are sporadic and nearly impossible to reproduce.

The common thread: None of these produce errors. They all produce wrong results that look right. The only reliable detection is testing from a spec that explicitly defines boundary behavior — not testing from the code's own logic.

Prevention at Each Governance Pillar

Design-time: Spec-first prompting with explicit boundary conditions. The spec IS the governance artifact. No spec, no generation.

Build-time: Test coverage requirements that mandate boundary cases. The AI Code Review Standard asks: "Does this handle boundaries correctly?" as question #2 of five.

People: The ability to look at plausible code and say "this is wrong" is exactly what we mean by "taste." It's the core skill of the AI-era developer — and it has to be cultivated through practice, not assumed.

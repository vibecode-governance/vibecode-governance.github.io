The Governance Model for AI-assisted Salesforce Development

Five pillars of control for Vibe Coding on Salesforce Platform

You can't govern vibe coding with a single checkpoint. You need controls at every stage of the lifecycle — from the moment the developer describes their intent, through the build pipeline, into production execution, across the data layer, and into the organizational practices that sustain it.

The failure taxonomy showed you what breaks. This page shows you where to put the controls. The model has five pillars, each addressing a different stage of the AI-assisted development lifecycle. Miss one pillar and at least one failure mode has a clear path to production.

The goal is not to slow down AI-assisted development. It's to make velocity safe by default. Each pillar makes the governed path the path of least resistance — so teams stay fast and stay compliant without extra effort.

Pillar 1: Design-Time Controls

Define the constraints before the AI writes anything.

This is the most overlooked pillar — and the one with the highest leverage. Design-time controls shape what the AI generates in the first place. If you get this right, you prevent failures at the source rather than catching them downstream.

The core principle: the human writes the specification, the AI writes the implementation. When developers skip the spec and jump straight to "build me a thing," the AI optimizes for plausibility. When they provide a structured spec with boundary conditions, the AI is constrained to generate something that matches.

Key Practices

Spec-Driven Prompting Define inputs, outputs, boundary conditions, and error states before the AI generates a single line. The spec becomes the acceptance test.
Test-First Generation Ask the AI to write tests from the spec before writing implementation. Forces alignment to requirements, not plausibility.
Prompt Templates with Security Defaults Organizational templates that pre-inject with sharing, USER_MODE, and security patterns so AI output is safe by default.
Constraint Definition Explicitly state what the code must NOT do. "Must not expose fields beyond the user's FLS." "Must not execute DML without access checks."

Addresses:

Logic Hallucination Security Bypass Comprehension Debt

Think of it this way: In traditional development, design review happens before coding begins. In vibe coding, the "design" IS the prompt. If your prompt has no constraints, your output has no guardrails. Making the prompt rigorous is equivalent to making the design rigorous.

On Salesforce: This is where developer-level System Prompts in Agentforce Builder come in. You can pre-load security requirements, naming standards, and architectural constraints directly into the agent's instruction set — so every code generation starts from a governed baseline.

Pillar 2: Build-Time Controls

Validate in the pipeline before it reaches production.

Design-time controls reduce the probability of bad output. Build-time controls catch what slips through. This is your CI/CD pipeline — the automated quality gate between "AI generated it" and "it's running in production."

The principle here: trust the output of the AI exactly as much as you'd trust a new junior hire's first PR — which is to say, validate everything before it ships. The pipeline doesn't care who (or what) wrote the code. It enforces the same standards regardless.

Key Practices

Salesforce Code Analyzer + DFA Static analysis with data-flow analysis in CI. Catches naked SOQL/DML, missing with sharing, hardcoded credentials, SSRF patterns.
Metadata Validation Rules Enforce naming conventions, ownership fields, description requirements. Reject Flows/triggers without fault paths or bulk-safety annotations.
AI Code Review Standard (5 Questions) Every PR that contains AI-generated code gets a structured review: Does it match the spec? Does it handle boundaries? Is it secure? Is it bulk-safe? Can the team maintain it?
Automation Overlap Detection Before deploying a new Flow or trigger, verify it doesn't conflict with existing automation on the same object. Map trigger order and identify potential race conditions.

Addresses:

Security Bypass State Corruption Logic Hallucination

The volume problem: AI-assisted teams can generate 3-5x more PRs per week. Your pipeline needs to handle this without becoming a bottleneck. The answer isn't less review — it's smarter, automated review. Code Analyzer catches 80% of security issues without a human reviewer needing to scan every line.

On Salesforce: The DevOps Center + Code Analyzer pipeline gives you an automated gate that enforces security and quality standards at deploy time. Shield's Event Monitoring adds a post-deployment audit trail so you can trace what was deployed, by whom, and when.

Pillar 3: Runtime Controls

Govern what agents and code do while they're executing.

Design and build controls cover code that humans deploy. Runtime controls cover what happens when agents execute autonomously. This is the most critical pillar for the agentic era — because once an agent can act, the gap between "misconfiguration" and "incident" shrinks to zero.

The principle: bounded autonomy. The agent can act within defined limits. Beyond those limits, it must escalate. The boundaries are enforced by the platform, not by the agent's good judgment.

Key Practices

Trust Layer (LLM Gateway) Every LLM call routes through a centralized gateway with injection defense, toxicity detection, PII masking, and a full audit trail. No direct model access.
Bounded Autonomy confirmationRequired on high-risk actions. Forced escalation paths. Action catalogs that whitelist what an agent can do. Hard spending/record limits.
Transaction Security Policies Real-time blocking rules that fire when agent behavior matches a threat pattern. Can block or require MFA before the action completes.
Event Monitoring & Anomaly Detection Continuous observation of agent activity. Surface unusual patterns: bulk data access, cross-object queries that don't match the agent's expected behavior, off-hours execution.

Addresses:

Prompt Injection & Agent Hijack Security Bypass State Corruption

The key shift: In traditional development, runtime monitoring is about availability — is the system up? In the agentic era, runtime monitoring is about intent — is the agent doing what we asked? This requires observable reasoning (seeing why the agent made a decision) and behavioral baselines (knowing what "normal" looks like for each agent role).

On Salesforce: The Trust Layer provides the gateway. Transaction Security Policies provide real-time blocking. Event Monitoring provides the audit trail. Together, they create a runtime governance layer that works whether the agent is a human-triggered Agentforce action or a fully autonomous background process.

Pillar 4: Data Controls

Protect what the AI reads, writes, and reasons over.

Code runs. Agents act. But what matters is the data. Every failure mode in the taxonomy ultimately manifests as a data problem: wrong values written (logic hallucination), unauthorized data exposed (security bypass), data exfiltrated (prompt injection), or records corrupted (state corruption). The data layer is your last line of defense.

The principle: enforce security at the data boundary, not in the application logic. If access control lives in the code layer, every new AI-generated method needs to re-implement it. If access control lives in the platform's data layer, it's enforced regardless of what code the AI generates.

Key Practices

USER_MODE Enforcement WITH USER_MODE on all queries, AccessLevel.USER_MODE on all DML. The database enforces the user's actual access — sharing rules, CRUD, FLS — regardless of what the code tries to do.
Platform Encryption (Shield) Encrypt sensitive fields at rest with tenant-controlled keys. Data remains protected even if unauthorized access occurs at the application layer.
Data Detect & Classification Automatically identify and classify sensitive data (PII, PHI, PCI) across the org. You can't protect what you can't find.
Field Audit Trail Retain field-level change history for up to 10 years. When an agent modifies data, you have a forensic record of what changed, from what value, and when.

Addresses:

Security Bypass Prompt Injection & Agent Hijack State Corruption

This is Salesforce's structural advantage: On general-purpose platforms, you build access control into your application. On Salesforce, access control is baked into the database layer. When you use USER_MODE, the platform enforces the user's permissions at the query level — the AI-generated code physically cannot bypass sharing rules, CRUD, or FLS. Security becomes a platform property, not an application responsibility.

On Salesforce: Shield (Platform Encryption + Event Monitoring + Field Audit Trail) provides the foundation. Data Detect classifies what's sensitive. Security Center gives you a single dashboard across all your orgs. Together they ensure that even if AI-generated code has a flaw, the data layer enforces the rules.

Pillar 5: People & Organizational Controls

Align roles, metrics, and incentives to the new reality.

Technology controls catch 80% of the failures. The remaining 20% require people who know what good looks like, metrics that measure the right things, and an organizational structure that rewards governance over raw velocity.

The principle: the developer's role shifts from "person who writes code" to "person who validates AI output against business intent." This isn't a demotion — it's an elevation. The cognitive work moves from syntax to judgment. But organizations need to explicitly build for this shift, or it doesn't happen.

Key Practices

New Metrics: TrueThroughput Measure value delivered, not volume produced. Lines of code and PRs merged mean nothing if the code gets reverted or creates incidents. Track: Code Turnover (<12%), Maintainability Index (>70), PR Cycle Time (<8hr).
Career Path Evolution Junior = Curator (evaluates AI output). Mid = Pipeline Owner (designs constraints). Senior = Constraint Architect (defines organizational policy). Promotion requires demonstrated "taste."
AI Code Review Standard Five questions per PR: Spec alignment? Boundary handling? Security enforcement? Bulk safety? Team maintainability? No PR ships without documented answers.
Interview & Hiring Redesign Three-stage assessment: Constraint definition (can they spec it?), Debugging hallucinations (can they catch the AI's errors?), Demonstrating taste (can they reject plausible-but-wrong output?).

Addresses:

Comprehension Debt Logic Hallucination

The "taste" concept: In a world where AI can generate any plausible-looking code, the critical skill is rejection — the ability to look at functionally correct output and say "this is wrong for our architecture, our scale, our security model." We call this "taste." It can't be automated. It must be cultivated through deliberate organizational practice.

How the Pillars Work Together

No single pillar is sufficient. Each addresses different failure modes at different stages. The model works because it's layered — a failure that slips past one pillar gets caught by the next.

Failure Mode Primary Pillar Backup Pillar Why Both
Logic Hallucination Design (spec-first) Build (test validation) Specs prevent wrong logic; tests catch what specs missed
Security Bypass Build (Code Analyzer) Data (USER_MODE) Static analysis catches most; platform enforcement catches all
Prompt Injection Runtime (Trust Layer) Data (least privilege) Detect injection at inference; limit damage through access control
State Corruption Build (metadata validation) Data (audit trail) Prevent bad automation from deploying; trace damage when it occurs
Comprehension Debt People (review standards) Design (spec requirement) People practices build understanding; specs make code traceable

Shared Responsibility: Salesforce secures the platform. You secure what you build on it. The five pillars tell you exactly where your responsibility lies — and where the platform already has you covered.

Why This Maps Naturally to Salesforce

Most platforms require you to build governance from scratch. On Salesforce, three of the five pillars already have built-in platform primitives:

Pillar Platform Primitive What It Gives You
Build Code Analyzer, DevOps Center, Metadata API Automated security scanning, deployment gates, metadata enforcement
Runtime Trust Layer, Transaction Security, Event Monitoring LLM gateway, real-time blocking, behavioral audit trail
Data Shield (Encryption + FAT + EM), Data Detect, Security Center Encryption at rest, 10-year audit, classification, cross-org visibility

The remaining two pillars (Design and People) require organizational investment — but the framework gives you a clear playbook for both. And because the platform pillars are already in place, you're not starting from zero.
